Install RubixKube on your **cloud Kubernetes cluster** to test our Site Reliability Intelligence platform in a cloud environment.

<Warning>
### Beta Release Notice
RubixKube is currently in **Beta**. While you can install it on cloud clusters for **testing and evaluation**, it is **NOT recommended for production environments** yet.

- **Safe for:** Development, staging, testing environments
- **Not ready for:** Production workloads, critical infrastructure
- **Coming soon:** Production-ready release with SLA guarantees
</Warning>

<Info>
### Why Install on Cloud?
Testing RubixKube on a cloud cluster gives you a more realistic experience compared to local KIND clusters, including real cloud resources, load balancers, and persistent storage.
</Info>

### Cloud Requirements

<Tabs>
  <Tab title="AWS EKS">
**Minimum Cluster Specs:**
    - Kubernetes version: 1.24+
    - Node type: t3.medium or larger (2 vCPU, 4GB RAM minimum)
    - Number of nodes: 1+ (2-3 recommended for testing)
    - IAM permissions for kubectl access

**Recommended for Testing:**
    - Node type: t3.large (2 vCPU, 8GB RAM)
    - Number of nodes: 2-3
    - Enable CloudWatch integration
  </Tab>

  <Tab title="Google GKE">
**Minimum Cluster Specs:**
    - Kubernetes version: 1.24+
    - Machine type: e2-medium or larger (2 vCPU, 4GB RAM minimum)
    - Number of nodes: 1+ (2-3 recommended for testing)
    - GKE cluster admin permissions

**Recommended for Testing:**
    - Machine type: e2-standard-2 (2 vCPU, 8GB RAM)
    - Number of nodes: 2-3
    - Enable GKE monitoring
  </Tab>

  <Tab title="Azure AKS">
**Minimum Cluster Specs:**
    - Kubernetes version: 1.24+
    - VM size: Standard_B2s or larger (2 vCPU, 4GB RAM minimum)
    - Number of nodes: 1+ (2-3 recommended for testing)
    - Azure RBAC permissions

**Recommended for Testing:**
    - VM size: Standard_D2s_v3 (2 vCPU, 8GB RAM)
    - Number of nodes: 2-3
    - Enable Azure Monitor integration
  </Tab>
</Tabs>

## Step 1: Create or Connect to Your Cluster

<Tabs>
  <Tab title="AWS EKS">
    ### Create an EKS Cluster (if you don't have one)

### Option 1: Using eksctl (Recommended for Testing)
    ```bash
    # Install eksctl (if not already installed)
    brew install eksctl  # macOS
    # For other OS: https://eksctl.io/installation/

    # Create cluster
    eksctl create cluster \
      --name rubixkube-test \
      --region us-west-2 \
      --nodes 2 \
      --node-type t3.medium \
      --version 1.28
    ```

    This takes **15-20 minutes** to complete.

### Option 2: Using AWS Console
    1. Go to [AWS EKS Console](https://console.aws.amazon.com/eks)
    2. Click **"Create cluster"**
    3. Name: `rubixkube-test`
    4. Kubernetes version: 1.24 or higher
    5. Follow the wizard to create node groups
    6. Wait for cluster to be active

    ### Connect to Your EKS Cluster

    ```bash
    # Configure kubectl to use your EKS cluster
    aws eks update-kubeconfig --region us-west-2 --name rubixkube-test
    ```

    ### Verify Connection

    ```bash
    kubectl get nodes
    ```

    Expected output:
    ```
    NAME                          STATUS   ROLES    AGE   VERSION
    ip-192-168-x-x.ec2.internal   Ready    <none>   5m    v1.28.x
    ip-192-168-y-y.ec2.internal   Ready    <none>   5m    v1.28.x
    ```

    <Tip>
**Cost Warning:** EKS costs ~$0.10/hour for the control plane + EC2 instance costs. Delete the cluster when done testing to avoid charges.
    </Tip>
  </Tab>

  <Tab title="Google GKE">
    ### Create a GKE Cluster (if you don't have one)

### Using gcloud CLI:
    ```bash
    # Set your project (replace with your project ID)
    gcloud config set project YOUR_PROJECT_ID

    # Create cluster
    gcloud container clusters create rubixkube-test \
      --zone us-central1-a \
      --num-nodes 2 \
      --machine-type e2-medium \
      --cluster-version latest
    ```

    This takes **5-10 minutes** to complete.

### Using Google Cloud Console:
    1. Go to [GKE Console](https://console.cloud.google.com/kubernetes)
    2. Click **"Create Cluster"**
    3. Choose **"Standard"** mode
    4. Name: `rubixkube-test`
    5. Zone: `us-central1-a`
    6. Number of nodes: 2
    7. Machine type: `e2-medium`
    8. Click **"Create"**
    ### Connect to Your GKE Cluster

    ```bash
    # Get credentials for kubectl
    gcloud container clusters get-credentials rubixkube-test --zone us-central1-a
    ```

    ### Verify Connection

    ```bash
    kubectl get nodes
    ```

    Expected output:
    ```
    NAME                              STATUS   ROLES    AGE   VERSION
    gke-rubixkube-test-pool-xxxxxx    Ready    <none>   3m    v1.28.x
    gke-rubixkube-test-pool-yyyyyy    Ready    <none>   3m    v1.28.x
    ```

    <Tip>
**Cost Warning:** GKE costs vary by region. A 2-node e2-medium cluster costs ~$50-70/month. Use `gcloud container clusters delete` to remove when done.
    </Tip>
  </Tab>

  <Tab title="Azure AKS">
    ### Create an AKS Cluster (if you don't have one)

### Using Azure CLI:
    ```bash
    # Login to Azure
    az login

    # Create resource group (if needed)
    az group create --name rubixkube-rg --location eastus

    # Create AKS cluster
    az aks create \
      --resource-group rubixkube-rg \
      --name rubixkube-test \
      --node-count 2 \
      --node-vm-size Standard_B2s \
      --enable-managed-identity \
      --generate-ssh-keys
    ```

    This takes **10-15 minutes** to complete.

### Using Azure Portal:
    1. Go to [Azure Portal](https://portal.azure.com)
    2. Search for **"Kubernetes services"**
    3. Click **"Create"** â†’ **"Kubernetes cluster"**

    5. Cluster name: `rubixkube-test`
    6. Region: East US
    7. Node count: 2
    8. Node size: Standard_B2s
    9. Click **"Review + Create"**
    ### Connect to Your AKS Cluster

    ```bash

    ```

    ```bash

    ```

    Expected output:
    ```
    NAME                                STATUS   ROLES   AGE   VERSION
    aks-nodepool1-xxxxxxxx-vmss000000   Ready    agent   4m    v1.28.x
    aks-nodepool1-xxxxxxxx-vmss000001   Ready    agent   4m    v1.28.x
    ```

    <Tip>
**Cost Warning:** AKS control plane is free, but you pay for VMs. A 2-node Standard_B2s cluster costs ~$60-80/month. Delete when done testing.

## Step 3: Install RubixKube Observer

### Run the Installation Command

Paste your command from the dashboard into your terminal:

```bash
kubectl apply -f "https://api.rubixkube.ai/install/observer.yaml?apiKey=rk_YOUR_API_KEY"
```

### What gets installed:
```
namespace/rubixkube created
serviceaccount/rubixkube-observer created
clusterrole.rbac.authorization.k8s.io/rubixkube-observer created
clusterrolebinding.rbac.authorization.k8s.io/rubixkube-observer created
deployment.apps/rubixkube-observer created
configmap/rubixkube-config created
secret/rubixkube-credentials created
```

<Accordion title=" What does the observer do?">
  The RubixKube observer agent:
  
  - **Monitors** your cluster resources (pods, deployments, services, nodes)
  - **Collects** metrics and events (NOT your application data)
  - **Detects** anomalies and potential issues
  - **Communicates** with RubixKube AI agents for analysis
  - **Reports** cluster health to your dashboard
  
**Privacy:** The observer only collects metadata and metrics - never your application logs or sensitive data.
</Accordion>

## Step 4: Verify Installation

Let's confirm RubixKube is running correctly in your cluster.

### Check Namespace Created

```bash
kubectl get namespaces | grep rubixkube
```

Expected output:
```
rubixkube   Active   30s
```

### Check Pod Status

```bash
kubectl get pods -n rubixkube
```

**Initial output** (pods starting):
```
NAME                                  READY   STATUS              RESTARTS   AGE
rubixkube-observer-7d4b9c8f6d-abc12   0/1     ContainerCreating   0          15s
```

**Wait 1-2 minutes**, then check again:

```bash
kubectl get pods -n rubixkube
```

**Success - Pod Running:**
```
NAME                                  READY   STATUS    RESTARTS   AGE
rubixkube-observer-7d4b9c8f6d-abc12   1/1     Running   0          2m
```

<AccordionGroup>
  <Accordion title=" What indicates successful installation?">
    Look for these indicators:
    
    | Column | Expected Value | Meaning |
    |--------|---------------|---------|
    | **READY** | `1/1` | Container is ready |
    | **STATUS** | `Running` | Pod is active |
    | **RESTARTS** | `0` or low number | Stable, no crashes |
    | **AGE** | Increasing | Pod staying alive |
    
    If you see `1/1 Running`, **you're good!**
  </Accordion>

    ```bash

    ```
