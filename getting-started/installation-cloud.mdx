Install RubixKube on your**cloud Kubernetes cluster**to test our Site Reliability Intelligence platform in a cloud environment.

<Warning>
### Beta Release Notice
RubixKube is currently in <strong>Beta</strong>. While you can install it on cloud clusters for <strong>testing and evaluation</strong>, it is <strong>NOT recommended for production environments</strong> yet.

- <strong>Safe for:</strong> Development, staging, testing environments
- <strong>Not ready for:</strong> Production workloads, critical infrastructure
- <strong>Coming soon:</strong> Production-ready release with SLA guarantees
</Warning>

<Info>
### Why Install on Cloud?
Testing RubixKube on a cloud cluster gives you a more realistic experience compared to local KIND clusters, including real cloud resources, load balancers, and persistent storage.
</Info>

<Tabs>

    - Number of nodes: 2-3
    - Enable GKE monitoring
  </Tab>

  <Tab title="Azure AKS">

    - VM size: Standard_B2s or larger (2 vCPU, 4GB RAM minimum)

## Step 1: Create or Connect to Your Cluster

<Tabs>
  <Tab title="AWS EKS">
    ### Create an EKS Cluster (if you don't have one)

### Option 1: Using eksctl (Recommended for Testing)
    ```bash
    # Install eksctl (if not already installed)
    brew install eksctl  # macOS
    # For other OS: https://eksctl.io/installation/

    # Create cluster
    eksctl create cluster \
      --name rubixkube-test \
      --region us-west-2 \
      --nodes 2 \
      --node-type t3.medium \
      --version 1.28
    ```

    This takes** 15-20 minutes**to complete.

### Option 2: Using AWS Console
    1. Go to [AWS EKS Console](https://console.aws.amazon.com/eks)
    2. Click**"Create cluster"** 3. Name: `rubixkube-test`
    4. Kubernetes version: 1.24 or higher
    5. Follow the wizard to create node groups
    6. Wait for cluster to be active

    ### Connect to Your EKS Cluster

    ```bash
    # Configure kubectl to use your EKS cluster
    aws eks update-kubeconfig --region us-west-2 --name rubixkube-test
    ```

    ### Verify Connection

    ```bash
    kubectl get nodes
    ```

    Expected output:
    ```
    NAME                          STATUS   ROLES    AGE   VERSION
    ip-192-168-x-x.ec2.internal   Ready    <none>   5m    v1.28.x
    ip-192-168-y-y.ec2.internal   Ready    <none>   5m    v1.28.x
    ```

    <Tip>
<strong>Cost Warning:</strong> EKS costs ~$0.10/hour for the control plane + EC2 instance costs. Delete the cluster when done testing to avoid charges.
    </Tip>
  </Tab>

## Step 3: Install RubixKube Observer

### Run the Installation Command

Paste your command from the dashboard into your terminal:

```bash
kubectl apply -f "https://api.rubixkube.ai/install/observer.yaml?apiKey=rk_YOUR_API_KEY"
```

### What gets installed:
```
namespace/rubixkube created
serviceaccount/rubixkube-observer created
clusterrole.rbac.authorization.k8s.io/rubixkube-observer created
clusterrolebinding.rbac.authorization.k8s.io/rubixkube-observer created
deployment.apps/rubixkube-observer created
configmap/rubixkube-config created
secret/rubixkube-credentials created
```

<Accordion title=" What does the observer do?">
  The RubixKube observer agent:
  
  - **Monitors**your cluster resources (pods, deployments, services, nodes)
  - **Collects**metrics and events (NOT your application data)
  - **Detects**anomalies and potential issues
  - **Communicates**with RubixKube AI agents for analysis
  - **Reports**cluster health to your dashboard
  
**Privacy:**The observer only collects metadata and metrics - never your application logs or sensitive data.
</Accordion>

## Step 4: Verify Installation

Let's confirm RubixKube is running correctly in your cluster.

### Check Namespace Created

```bash
kubectl get namespaces | grep rubixkube
```

Expected output:
```
rubixkube   Active   30s
```

### Check Pod Status

```bash
kubectl get pods -n rubixkube
```

**Initial output**(pods starting):
```
NAME                                  READY   STATUS              RESTARTS   AGE
rubixkube-observer-7d4b9c8f6d-abc12   0/1     ContainerCreating   0          15s
```

**Wait 1-2 minutes**, then check again:

```bash
kubectl get pods -n rubixkube
```

**Success - Pod Running:**```
NAME                                  READY   STATUS    RESTARTS   AGE
rubixkube-observer-7d4b9c8f6d-abc12   1/1     Running   0          2m
```

<AccordionGroup>
  <Accordion title=" What indicates successful installation?">
    Look for these indicators:
    
    | Column | Expected Value | Meaning |
    |--------|---------------|---------|
    |**READY**| `1/1` | Container is ready |
    |**STATUS**| `Running` | Pod is active |
    |**RESTARTS**| `0` or low number | Stable, no crashes |
    |**AGE**| Increasing | Pod staying alive |
    
    If you see `1/1 Running`,**you're good!**  </Accordion>

  <Accordion title=" Troubleshooting: ImagePullBackOff">
**Problem:**Pod can't download the RubixKube image.
    
**Common causes:**    - Cluster lacks internet access
    - Container registry authentication issue
    - Network policies blocking outbound traffic
    
### Solutions:
    1.**Check pod details:**    ```bash
    kubectl describe pod -n rubixkube -l app=rubixkube-observer
    ```
    
    2.**Verify internet connectivity:**```bash
    kubectl run test-curl --image=curlimages/curl -i --rm -- curl -I https://api.rubixkube.ai
    ```
    
    3.**Check events:**    ```bash
    kubectl get events -n rubixkube --sort-by='.lastTimestamp'
    ```
    
    If issue persists after 5 minutes, contact support@rubixkube.ai
  </Accordion>

  <Accordion title=" Troubleshooting: CrashLoopBackOff">
**Problem:**Pod starts but immediately crashes.
    
**Most common cause:**Invalid or missing API key
    
### Solution:
    1.**Check logs to see error:**```bash
    kubectl logs -n rubixkube deployment/rubixkube-observer
    ```
    
    2.**Look for authentication errors:**    - "Invalid API key"
    - "Unauthorized"
    - "Failed to connect to RubixKube API"
    
    3.**Reinstall with correct API key:**    ```bash
    # Delete existing installation
    kubectl delete namespace rubixkube
    
    # Wait 30 seconds
    sleep 30
    
    # Get fresh command from console.rubixkube.ai and run it
    ```
  </Accordion>

  <Accordion title=" Troubleshooting: Pending Status">
**Problem:**Pod stuck in "Pending" state.
    
**Common causes:**- Not enough cluster resources
    - No nodes available
    - Storage provisioning issues
    
### Solutions:
    1.**Check node resources:**```bash
    kubectl top nodes
    kubectl describe nodes
    ```
    
    2.**Check pod events:**    ```bash
    kubectl describe pod -n rubixkube -l app=rubixkube-observer
    ```
    
    3.**Scale your node pool**if resources are exhausted
  </Accordion>
</AccordionGroup>

### Check Observer Logs

View what RubixKube is doing:

```bash
kubectl logs -n rubixkube deployment/rubixkube-observer --tail=50
```

### Healthy logs should show:

2. Navigate to**Dashboard**or**Clusters**page

## Uninstalling RubixKube

### Remove RubixKube from Your Cluster

If you want to uninstall RubixKube:

```bash
# Delete the entire RubixKube namespace
kubectl delete namespace rubixkube

This removes:
- Observer deployment
- Service accounts
- RBAC roles and bindings
- Configuration and secrets

## Common Issues & Solutions

<AccordionGroup>
  <Accordion title="kubectl: connection refused">
**Error:**`The connection to the server localhost:8080 was refused`
    
**Cause:**kubectl is not configured to connect to your cluster
    
**Solution:**- Run the "update kubeconfig" command for your cloud provider (see Step 1)
    - Verify with: `kubectl config current-context`
  </Accordion>

  <Accordion title="Unauthorized / Forbidden errors">
**Error:**`Error from server (Forbidden): pods is forbidden`
    
**Cause:**Your user doesn't have proper RBAC permissions
    
### Solution:
**For EKS:**```bash
    # Ensure you're using cluster creator credentials
    aws sts get-caller-identity
    ```
    
**For GKE:**    ```bash
    # Grant yourself cluster-admin
    kubectl create clusterrolebinding cluster-admin-binding \
      --clusterrole=cluster-admin \
      --user=$(gcloud config get-value core/account)
    ```
    
**For AKS:**```bash
    # Get admin credentials
    az aks get-credentials --resource-group rubixkube-rg --name rubixkube-test --admin
    ```
  </Accordion>

  <Accordion title="Observer pod keeps restarting">
**Problem:**RESTARTS column shows increasing numbers
    
**Diagnosis:**    ```bash
    # Check logs for errors
    kubectl logs -n rubixkube deployment/rubixkube-observer --previous
    
    # Check events
    kubectl get events -n rubixkube --sort-by='.lastTimestamp'
    ```
    
**Common causes:**- Network connectivity issues
    - API key expired or invalid
    - Memory limits too low
    
**Solution:**Contact support@rubixkube.ai with the logs
  </Accordion>

  <Accordion title="Cluster not appearing in dashboard">
**Problem:**Cluster successfully installed but not showing in RubixKube console
    
### Checklist:
    1.**Wait 2-3 minutes**- Initial sync takes time
    2.**Check pod is Running:**```bash
    kubectl get pods -n rubixkube
    ```
    
    3.**Check observer logs for connection:**    ```bash
    kubectl logs -n rubixkube deployment/rubixkube-observer | grep -i "connect"
    ```
    
    4.**Verify API key is correct:**```bash
    kubectl get secret -n rubixkube rubixkube-credentials -o yaml
    ```
    
    5.**Check outbound connectivity:**    ```bash
    kubectl run test-curl --image=curlimages/curl -i --rm -- \
      curl -v https://api.rubixkube.ai/health
    ```
  </Accordion>

  <Accordion title="Network policy blocking RubixKube">
**Problem:**Cluster has strict network policies
    
**Solution:**Allow egress to RubixKube API:
    
    ```yaml
    apiVersion: networking.k8s.io/v1
    kind: NetworkPolicy
    metadata:
      name: allow-rubixkube-egress
      namespace: rubixkube
    spec:
      podSelector:
        matchLabels:
          app: rubixkube-observer
      policyTypes:
      - Egress
      egress:
      - to:
        - namespaceSelector: {}
        ports:
        - protocol: TCP
          port: 443
      - to:
        - podSelector: {}
    ```
    
    Apply with:
    ```bash
    kubectl apply -f network-policy.yaml
    ```
  </Accordion>
</AccordionGroup>

## Resource Limits & Cost Optimization

### Set Resource Limits (Recommended)

To control RubixKube observer resource usage:

```yaml
# Download the observer manifest
curl -o observer.yaml "https://api.rubixkube.ai/install/observer.yaml?apiKey=rk_YOUR_API_KEY"

# Edit the deployment section to add resource limits
# Then apply:
kubectl apply -f observer.yaml
```

**Example limits:**```yaml
resources:
  requests:
    cpu: 100m
    memory: 256Mi
  limits:
    cpu: 500m
    memory: 512Mi
```

### Cost Management Tips

<Accordion title="AWS Cost Optimization">
  - Use**Spot Instances**for node groups (not for control plane)
  - Set up**Cluster Autoscaler**to scale down when idle
  - Use**Fargate**for serverless nodes (optional)
  - Enable**Cost Explorer**to track spending
  -**Delete cluster**when not in use: `eksctl delete cluster`
</Accordion>

<Accordion title="GKE Cost Optimization">
  - Enable**Autopilot**mode for cost-efficient scaling
  - Use**Preemptible VMs**for testing (cheaper)
  - Set**node auto-provisioning**for dynamic sizing
  - Enable**GKE usage metering**to track costs
  -**Delete cluster**when not testing: `gcloud container clusters delete`
</Accordion>

<Accordion title="Azure Cost Optimization">
  - Use**Spot VMs**for node pools (70-90% cheaper)

## Security Considerations

<AccordionGroup>
  <Accordion title="What data does RubixKube collect?" icon="shield">
    RubixKube observer collects:
    
**Metadata:**Pod names, namespaces, labels
**Metrics:**CPU, memory, network usage
**Events:**Kubernetes events and state changes
**Status:**Health checks and readiness
    
**NEVER collects:**Application logs, secrets, environment variables, or business data
    
    All data is encrypted in transit (TLS) and at rest.
  </Accordion>

  <Accordion title="What permissions does RubixKube need?" icon="key">
    RubixKube observer requires**read-only**cluster access:
    
    -**View pods, deployments, services**(read-only)
    -**Read events and metrics**(read-only)
    -**No write/delete permissions**on your resources
    
    View exact permissions:
    ```bash
    kubectl get clusterrole rubixkube-observer -o yaml
    ```
  </Accordion>

  <Accordion title="Is my API key secure?" icon="lock">
    Your API key is stored as a Kubernetes secret:
    
    ```bash
    kubectl get secret -n rubixkube rubixkube-credentials -o yaml
    ```
    
**Best practices:**    - Never commit API keys to git
    - Rotate keys every 90 days (from dashboard)
    - Revoke keys immediately if compromised
    - Use different keys for different environments
  </Accordion>
</AccordionGroup>
