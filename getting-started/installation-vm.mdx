## Why Monitor Linux VMs with RubixKube?

Running the RubixKube Observer on Linux VMs gives you direct system-level visibility:
- **CPU metrics** - Usage, load averages, per-core utilization
- **Memory metrics** - RAM usage, swap, available/cached memory
- **Disk metrics** - Per-mount usage, free space, I/O operations and latency
- **Network metrics** - Interface statistics, errors, dropped packets
- **Process metrics** - Top CPU/memory consumers, zombie processes, context switches
- **System events** - Service restarts, kernel errors, package updates

## What Gets Installed

The installer will:
1. Download the observer binary (optimized for VM monitoring)
2. Create a systemd service: `rubixkube-observer`
3. Configure environment variables (API key, polling intervals, thresholds)
4. Create log directory: `/var/log/rubixkube/`
5. Start the service automatically on boot

### Installation Paths

<Tabs>
  <Tab title="Standard Installation (/opt)">
    Binary installed to `/opt/rubixkube/observer`

    Config: `/etc/rubixkube/observer.env`

    Logs: `/var/log/rubixkube/observer.log`
  </Tab>

    ```bash
      --api-key=rk_YOUR_API_KEY \
      --install-dir=/custom/path
    ```

## Configuration

### Environment Variables

All configuration uses environment variables. The installer sets these automatically, but you can modify them:

```bash
sudo nano /etc/rubixkube/observer.env
sudo systemctl restart rubixkube-observer
```

**Key variables:**

| Variable | Default | Description |
|----------|---------|-------------|
| `API_KEY` | - | Your RubixKube API key (required) |
| `CLUSTER_TYPE` | `vm` | Environment mode |
| `TRANSPORT_TYPE` | `nats` | Transport: `nats` or `ws` |
| `SYSTEM_POLL_INTERVAL` | `30s` | How often to collect metrics |
| `INSIGHTS_ENABLED` | `true` | Publish insights to NATS |
| `CPU_THRESHOLD` | `85` | CPU % to trigger alert |
| `MEMORY_THRESHOLD` | `85` | Memory % to trigger alert |
| `DISK_THRESHOLD` | `90` | Disk % to trigger alert |
| `LOG_LEVEL` | `info` | Log verbosity: `debug`, `info`, `warn`, `error` |

### Polling Intervals

Adjust how frequently the observer collects metrics:

```bash
# Collect every 15 seconds (high frequency)
SYSTEM_POLL_INTERVAL=15s

# Collect every 60 seconds (low frequency, lower overhead)
SYSTEM_POLL_INTERVAL=60s
```

<Tip>
Default 30s is good for most environments. Use 15s for fast-changing workloads, 60s+ for stable infrastructure to reduce overhead.
</Tip>

### Alert Thresholds

Customize when alerts trigger:

```bash
sudo tee /etc/rubixkube/observer.env << EOF
CPU_THRESHOLD=75          # Alert at 75% CPU
MEMORY_THRESHOLD=80       # Alert at 80% memory
DISK_THRESHOLD=85         # Alert at 85% disk
CONTEXT_SWITCH_THRESHOLD=50000
EOF

sudo systemctl restart rubixkube-observer
```

### Custom Logging

Enable debug logging for troubleshooting:

```bash
echo "LOG_LEVEL=debug" | sudo tee -a /etc/rubixkube/observer.env
sudo systemctl restart rubixkube-observer

# View debug logs
sudo journalctl -u rubixkube-observer -f
```

```bash

sudo systemctl restart rubixkube-observer
```

```bash

sudo journalctl -u rubixkube-observer -n 50

sudo journalctl -u rubixkube-observer -f

sudo journalctl -u rubixkube-observer --since "2 hours ago"
```

## Troubleshooting

### Service won't start

Check logs for errors:
```bash
sudo journalctl -u rubixkube-observer -n 100
```

Common issues:
- **"Invalid API key"** ŌĆö Verify key in `observer.env`
- **"Permission denied"** ŌĆö Run with sudo
- **"Port already in use"** ŌĆö Check if another observer is running: `sudo lsof -i :4222`

### Metrics not appearing

1. Verify observer is running: `sudo systemctl status rubixkube-observer`
2. Check connectivity: `curl -I https://api.rubixkube.ai`
3. Wait 1-2 minutes for first metrics
4. Verify environment appears in dashboard

### High CPU/Memory usage

The observer is typically lightweight (~50MB memory, `<5%` CPU). If high:

1. Increase `SYSTEM_POLL_INTERVAL` (default: 30s)
   ```bash
   echo "SYSTEM_POLL_INTERVAL=60s" | sudo tee -a /etc/rubixkube/observer.env
   sudo systemctl restart rubixkube-observer
   ```

2. Reduce log level: Set `LOG_LEVEL=warn`

3. Check for runaway processes: `ps aux | grep observer`

### Network connectivity issues

Verify outbound access:
```bash
# Test connectivity to RubixKube
curl -I https://api.rubixkube.ai

# Test DNS
nslookup api.rubixkube.ai

# Check firewall
sudo iptables -L -n | grep 443
```

## Uninstalling

To remove the RubixKube observer:

```bash
curl -fsSL https://api.rubixkube.ai/install/observer.sh | bash -s -- \
  --api-key=rk_YOUR_API_KEY \
  --uninstall
```

Or manually:
```bash
sudo systemctl stop rubixkube-observer
sudo systemctl disable rubixkube-observer
sudo rm /etc/systemd/system/rubixkube-observer.service
sudo rm -rf /opt/rubixkube
sudo rm -rf /var/log/rubixkube
sudo systemctl daemon-reload
```

## Monitoring Multiple VMs

To monitor multiple VMs, run the installer on each VM:

```bash
# VM 1
ssh vm1.example.com
curl -fsSL https://api.rubixkube.ai/install/observer.sh | bash -s -- --api-key=rk_YOUR_API_KEY

# VM 2
ssh vm2.example.com
curl -fsSL https://api.rubixkube.ai/install/observer.sh | bash -s -- --api-key=rk_YOUR_API_KEY
```

Each observer registers as a separate environment in RubixKube, identified by its hostname.
